## ❓ 第六轮问题：上下文管理

### Q40. 上下文压缩策略

**当上下文超过限制时，如何压缩？**

**方案 A - 滑动窗口**:
```
保留: 最近 N 轮对话 + 系统提示
删除: 最早的对话
```

**方案 B - 摘要压缩**:
```
1. 提取早期对话的关键信息
2. 生成摘要（如: "前10轮讨论了剧情走向，决定采用方案A"）
3. 用摘要替换原对话
4. 保留最近对话原文
```

**方案 C - 分层保留**:
```
1. 决策类消息（如选择剧情走向）→ 始终保留
2. 思考过程 → 可压缩为摘要
3. 普通对话 → 滑动窗口
```

**你的选择是？**C，如果可能可以参考OMO的自动压缩方案

### Q41. 上下文估算

**如何估算当前上下文长度？**

- **A.** 按字符数估算（1 token ≈ 1.5 字符中文）
- **B.** 使用 tokenizer 精确计算
- **C.** 由后端实时计算并推送
- **D.** 显示估算值，实际由 LLM API 报错时处理

**你的选择是？**A，接近200K的上下文长度90%就自动压缩，因为GLM5最大接受200K上下文
