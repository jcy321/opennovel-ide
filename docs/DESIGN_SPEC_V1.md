## 一、核心架构设计

### 1.1 三种使用端模式

用户可通过不同启动参数选择不同的使用方式：

| 启动命令 | 模式 | 说明 | 监听端口 |
|---------|------|------|---------|
| `opennovel` | IDE模式 (默认) | 使用 opennovel-ide 桌面客户端连接 | 6688 |
| `opennovel -web` | Web模式 | 浏览器访问 http://服务器地址:端口 | 80/443 |
| `opennovel -ext` | 扩展模式 | 原版 VS Code + opennovel 插件连接 | 6689 |

**架构示意**:
```
┌─────────────────────────────────────────────────────────────┐
│                    部署机 (后端 opencore)                    │
│                                                             │
│   ┌─────────────────────────────────────────────────────┐  │
│   │              opennovel 后端服务                      │  │
│   │  ┌─────────┐  ┌─────────┐  ┌─────────────────────┐  │  │
│   │  │ IDE端口 │  │ Web端口 │  │ Extension端口       │  │  │
│   │  │  :6688  │  │  :80    │  │  :6689              │  │  │
│   │  └────┬────┘  └────┬────┘  └─────────┬───────────┘  │  │
│   └───────┼────────────┼─────────────────┼──────────────┘  │
│           │            │                 │                  │
└───────────┼────────────┼─────────────────┼──────────────────┘
            │            │                 │
   ┌────────┴───┐  ┌─────┴─────┐   ┌──────┴──────┐
   │ opennovel  │  │  浏览器   │   │ VS Code +   │
   │ -ide 客户端│  │  (Web)    │   │ opennovel   │
   │ (Electron) │  │           │   │ 插件        │
   └────────────┘  └───────────┘   └─────────────┘
```

### 1.2 商业定位

- 灵感来源：2026年初 OpenClaw 快速火爆但部署难度高
- 商业价值：降低部署门槛，提前布局 LLM tokens 消费市场
- 三端并行：覆盖不同技术能力的用户群体

### 1.3 后端技术要求

**需要实现的功能**：
1. 文件系统操作（读写书籍数据）
2. 群聊消息流（WebSocket/SSE）
3. Agent 状态同步
4. 多端口监听（IDE/Web/Extension）

**建议技术栈**：
- 保持现有 Axum (Rust) 后端
- 新增端口配置模块
- WebSocket 支持双向通信
- HTTP API 供 Web 和 Extension 模式使用

## 十一、群聊界面详细设计（第二轮确认）

### 11.1 消息显示格式

采用**紧凑式**设计（类似 QQ/微信），思考过程位于消息上方：

```
🟢 天道 14:32
💭 [点击展开思考过程]

我推演了第三章的三种可能走向...
推荐走向: [A] [B] [C]
────────────────────────────────
```

### 11.2 思考过程展示逻辑

| 阶段 | 行为 |
|------|------|
| Agent 开始回复 | 思考过程默认**展开** |
| 收到第一个结果字后 | 自动**折叠**思考过程 |
| 用户操作 | 可随时手动展开/折叠 |

### 11.3 批注存储格式

采用 **Markdown + HTML 注释** 方式存储：

```markdown
# 第三章 初入修仙界

李明站在青云宗的山门前，望着云雾缭绕的山峰...

<!-- 
  批注: 天道
  时间: 2026-03-19 14:32
  类型: 建议
  内容: 建议在此增加环境描写，突出青云宗的仙气氛围
  状态: 待处理
-->

...
```

**优势**：
- 兼容所有 Markdown 编辑器
- 注释在渲染后不可见
- 支持脚本批量转换为 Word/PDF
- 用户可 Ctrl+点击跳转到源文件

### 11.4 批注交互流程

1. Agent 在群聊回复中说明批注理由
2. 给出被批注文件的完整路径
3. 用户 Ctrl+点击跳转到该 Markdown 文件
4. 用户在群聊中 @对应 Agent 讨论想法
5. 达成一致后手动编辑文件

## 二十九、完整工作流程设计（第四轮确认 - 核心设计）

### 29.1 核心理念澄清

**天道不是任务编排器**，天道是剧情推演 Agent。

各 Agent 职责：
| Agent | 职责 | 维护的知识库 |
|-------|------|-------------|
| 规划者 | 阶段一与用户讨论书籍设定 | - |
| 天道 | 推演后续剧情、维护伏笔 | 伏笔数据库 |
| 刘和平 | 人物一致性检查、人物弧光分析 | 人物数据库 |
| 世界观守护者 | 世界观一致性检查 | 地图数据库、世界观数据库 |
| 执笔 | 撰写章节内容 | - |
| 观察者 | 更新知识库 | 各知识库（被调用时） |

### 29.3 Agent 协作串行流程

撰写一个新章节时的 Agent 协作顺序：

```
天道推演 → 刘和平审阅（人物） → 世界观守护者审阅 → 天道调整 → 用户确认 → 执笔撰写 → 知识库更新
```

**关键特点**：
- **串行执行**：每个 Agent 依次处理，上一个完成后交给下一个
- **批注传递**：每个 Agent 在文件中添加批注和签名
- **天道协调**：天道负责收集所有批注，做出最终调整决策
- **用户确认**：用户是最终决策者，不满意可要求调整

## 三十一、批注处理详细设计（第四轮确认）

### 31.1 批注格式

批注写入对应文字之后：

```markdown
李明站在青云宗的山门前，望着云雾缭绕的山峰...

<!-- 批注: 刘和平 | 审阅 | 此处可增加李明的内心独白，体现其初入仙门的紧张 | 2026-03-19 14:32 -->

天空中飘荡着几朵白云...

<!-- 批注: 世界观守护者 | 审阅 | 符合青云宗设定，无冲突 | 2026-03-19 14:35 -->
```

### 31.2 批注处理流程

```
1. Agent 创建批注（写入对应部分之后）
2. 批注依次传递给下一个 Agent
3. 最后一个批注的 Agent @用户请示裁决
4. 用户决定：
   - 直接采用
   - @对应 Agent 回复具体想法
5. 处理完成后，批注保留在文件中（作为历史记录）
```

### 31.3 批注冲突处理

**采用用户裁决方案**：
- 当多个 Agent 对同一段文字给出不同建议时
- 由最后一个批注的 Agent @用户请示裁决
- 用户可以直接采用，也可以 @对应 Agent 讨论具体想法
